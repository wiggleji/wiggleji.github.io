# 무엇을 설명하는가

서버에 트래픽이 폭증할 때, **시스템을 보호하면서 유저에게 공정한 경험을 제공하는 스로틀링 전략**을 세워야하는 상황이 발생한다.

이번 글에서는 이런 스로틀링 전략을 정의와 특징부터<br />
적용가능한 시나리오를 구체적으로 다뤄본다:

- Rate Limiting — 요청을 거부하여 시스템을 보호하는 가장 기본적인 접근 (Fixed Window, Sliding Window, Sliding Window Counter)
- Token Bucket — 순간 버스트를 허용하면서 평균 처리량을 제한하는 유연한 방식
- Queue-based Throttling — 거부 대신 줄을 세워 유저에게 순번과 예상 대기 시간을 제공하는 방식
- 세 가지 접근의 트레이드오프 비교와 상황별 선택 기준
- Redis 기반 대기열 시스템의 실전 설계 (Sorted Set, 입장 토큰, 마이크로 배칭)

# 왜 작성하는가

누구나 한번쯤 비슷한 UI 를 본 경험이 있을 것이다.

서버를 보호하는 방법은 여러 가지가 있지만, **"보호"만 생각하면 유저 경험이 희생**된다. "나중에 다시 시도하세요"를 반환하면 시스템은 살아남지만, 유저는 떠나거나 새로고침을 반복하며 오히려 트래픽을 악화시킬 수 있다.

# 서론

## 서버는 무한하지 않다

서버에는 물리적인 한계가 있다. DB 커넥션 풀, 스레드 풀, 메모리, 외부 API 호출 한도 — 당연하지만 모두 물리적으로 유한한 자원이다.

평소 사용량으로 보면 전혀 문제가 없다. 초당 100건 정도의 요청이 들어오고, 대부분 DB 커넥션 풀 50개 정도면 충분히 감당할 수 있다.

그런데 블랙 프라이데이나 서비스 이용자가 갑자기 증가하게 되면...?

```
평상시:   ████░░░░░░ (40% 사용)

스파이크: ████████████████████████████████ (3,200% 사용)
```

초당 10,000건이 밀려온다고 하자. DB 커넥션 풀 50개로는 초당 250건이 한계다. 나머지 9,750건은 커넥션을 얻기 위해 대기하다 타임아웃된다.

```
유저: "결제 버튼 눌렀는데 아무 반응이 없네?" → 새로고침!
유저: "아직도 안 되네?" → 새로고침! 새로고침!
```

타임아웃된 유저가 재시도하면 트래픽이 더 늘어난다. 이 **재시도 폭풍(Retry Storm)**은 서버를 완전히 멈추게 한다.<br />
한 서비스의 장애가 다른 서비스로 전파되는 **Cascading Failure**까지 이어질 수 있다.<br />

기술의 오류는 서비스 장애로 이어지고, 결과적으로 사용자는 이탈하게 되어 회사 매출에 큰 타격을 받을 수 있는 상황이 발생한다.

이러한 상황을 막는 방법은 크게 두 가지 관점으로 정의할 수 있다:

1. **거부**: 처리할 수 없는 요청을 빠르게 거부한다 → Rate Limiting, Token Bucket
2. **대기**: 요청을 줄 세워서 순서대로 처리한다 → Queue-based Throttling

어떤 걸 쓸지는 **"유저가 기다릴 의사가 있는가"** 에 따라 결정된다. 이 기준을 중심으로 각 전략을 살펴보자.

# 본론

## 1. Rate Limiting: 초과 요청 즉시 거부

가장 단순한 접근이다. 단위 시간당 허용 요청 수를 정하고, 초과하면 `429 Too Many Requests`를 반환한다.

"1초에 100건까지만 받겠다. 나머지는 돌아가라."

### 1-1. Fixed Window Counter

```
[0초 ─────── 1초][1초 ─────── 2초]
 ██████████ (10/10)  ███░░░░░░░ (3/10)
         ↑ 한도 도달, 이후 거부
```

1초 단위로 카운터를 두고, 한도에 도달하면 거부한다. Redis라면 `INCR` + `EXPIRE`로 간단하게 구현할 수 있다.

```
INCR rate:{userId}      → 카운터 증가
EXPIRE rate:{userId} 1  → 1초 후 리셋
```

간단하고 직관적이지만, 치명적인 약점이 하나 있다.

**경계 폭발(Boundary Burst)**

<img src="/images/20260402_throttling/boundary_burst.svg" width="640" />

0.9초에 10건을 보내고, 1.0초에 윈도우가 리셋되면서 다시 10건을 보낸다. 0.2초 사이에 20건이 통과한다.<br />
한도가 10/s인데 순간적으로 그 2배가 들어오는 것이다. 윈도우의 경계(boundary)에서 제한이 무너진다.

### 1-2. Sliding Window Log

경계 폭발을 해결하기 위해 **각 요청의 타임스탬프를 기록**하고, 현재 시각 기준으로 윈도우 내 요청 수를 센다.

```
현재: t=1.3s
윈도우: [0.3s ─────── 1.3s]
        t=0.5  t=0.8  t=1.0  t=1.2 → 4건 (한도 10 이내 → 허용)
```

윈도우가 요청 시각 기준으로 슬라이딩(이동)하기 때문에 경계 폭발이 발생하지 않는다. 정확하다.

그런데 대가가 있다. **모든 요청의 타임스탬프를 저장해야 한다**. Redis Sorted Set으로 구현할 수 있지만(`ZADD` + `ZRANGEBYSCORE`), 트래픽이 많을수록 Set 크기가 비례해서 커진다. 초당 10,000건이면 윈도우 1초에 10,000개의 타임스탬프를 유지해야 한다.

### 1-3. Sliding Window Counter (하이브리드)

Fixed Window의 효율성과 Sliding Window의 정확성을 절충한 방식이다.

```
현재 윈도우:     [──────── 60% 경과 ────────]
이전 윈도우 가중치: 40% × 이전 카운트
현재 윈도우 카운트: 100% × 현재 카운트
예상 요청 수 = (이전 × 0.4) + 현재
```

아이디어는 간단하다. "이전 윈도우의 남은 비율"을 현재 윈도우에 가중치로 합산한다.<br />
Redis 키 2개(이전 카운트, 현재 카운트)만 있으면 되므로 메모리 효율적이면서 경계 폭발을 상당 부분 완화한다.

완벽하게 정확하지는 않지만, 실무에 적용할 수 있는 적합한 방법 중 하나라고 볼 수 있다.

### 1-4. Rate Limiting의 한계

어떤 알고리즘을 쓰든 Rate Limiting의 본질은 같다: **초과 요청을 거부한다.**

```
유저 → [Rate Limiter] → 429 "나중에 다시 시도하세요"
                          ↓
                   유저: 새로고침! (재시도)
                          ↓
                   트래픽 더 증가
```

시스템 보호에는 효과적이지만, 유저 경험에는 치명적이다. 특히 블랙 프라이데이같이 유저가 "지금 아니면 안 돼!"라고 생각하는 상황에서 429를 반환하면:

- 유저는 **언제** 다시 시도해야 하는지 모른다
- 재시도 → 트래픽 증가 → 더 많은 429 → 악순환
- **공정성 부재**: 먼저 온 사람이나 나중에 온 사람이나 운에 의존

> **Rate Limiting이 적합한 시나리오**<br />
> API 남용 방지, Bot/크롤러 차단, 공개 API 사용량 제한(GitHub API, OpenAI API) 등 — **유저가 거부를 "정상"으로 받아들이는 상황**

## 2. Token Bucket: 버스트를 허용하는 유연한 제한

Rate Limiting이 "벽"이라면, Token Bucket은 "수도꼭지"다.<br />
버킷에 토큰이 일정 속도로 채워지고, 요청마다 토큰을 하나씩 소비한다.<br />
버킷에 토큰이 남아 있으면 요청이 통과하고, 비어 있으면 거부한다.

### 2-1. 동작 원리

<img src="/images/20260402_throttling/token_bucket.svg" width="640" />

핵심 파라미터는 두 개다:

- **버킷 용량(Capacity)**: 한 번에 허용할 최대 버스트 크기
- **보충 속도(Refill Rate)**: 초당 채워지는 토큰 수 = 장기 평균 허용 TPS

Rate Limiting과의 결정적 차이는 **버스트 허용**이다. Fixed Window에서 한도 10이면 절대 10을 넘길 수 없지만,<br />
Token Bucket은 버킷에 쌓여 있던 토큰만큼 순간적으로 한도를 초과할 수 있다.

| 특성        | Fixed Window       | Token Bucket                      |
| ----------- | ------------------ | --------------------------------- |
| 버스트 허용 | 불가 (한도 = 최대) | 가능 (버킷에 쌓인 만큼)           |
| 평균 처리량 | 윈도우 크기에 종속 | Refill Rate로 결정                |
| 구현 복잡도 | 낮음 (카운터 1개)  | 중간 (토큰 수 + 마지막 보충 시각) |
| 유저 경험   | 초과 시 즉시 거부  | 여유가 있으면 통과                |

### 2-2. Redis Lua로 원자적 구현

Token Bucket의 핵심은 "토큰 수 확인 → 차감"이 원자적이어야 한다는 것이다.<br />
두 요청이 동시에 "토큰 1개 남음"을 읽고 둘 다 통과하면 안 된다.<br />
Redis의 경우, Lua 스크립트로 이를 보장할 수 있다:

```lua
-- KEYS[1] = bucket:{clientId}
-- ARGV[1] = capacity, ARGV[2] = refill_rate(/s), ARGV[3] = now(ms)

local bucket = redis.call('HMGET', KEYS[1], 'tokens', 'last_refill')
local tokens = tonumber(bucket[1]) or tonumber(ARGV[1])
local last   = tonumber(bucket[2]) or tonumber(ARGV[3])

-- 경과 시간만큼 토큰 보충
local elapsed = (tonumber(ARGV[3]) - last) / 1000
local refilled = math.min(
    tonumber(ARGV[1]),
    tokens + elapsed * tonumber(ARGV[2])
)

if refilled >= 1 then
    redis.call('HMSET', KEYS[1],
        'tokens', refilled - 1,
        'last_refill', ARGV[3])
    redis.call('EXPIRE', KEYS[1], math.ceil(tonumber(ARGV[1]) / tonumber(ARGV[2])) + 1)
    return 1  -- 허용
else
    return 0  -- 거부
end
```

Lua 스크립트는 Redis에서 원자적으로 실행되므로 Race condition이 발생하지 않는다.

### 2-3. 그래도 결국은 거부한다

Token Bucket은 Rate Limiting보다 유연하지만, 본질은 같다: **초과 요청을 거부한다**.<br />
블랙 프라이데이에 1만 명이 동시에 접속하면, 버킷 용량이 아무리 커도 대부분은 거부당한다. 유저는 여전히 "언제 다시 해야 하지?"를 모른다.

> **Token Bucket이 적합한 시나리오**<br />
> API Gateway의 서비스별 트래픽 쉐이핑, 마이크로서비스 간 호출 보호, 짧은 버스트를 허용해야 하는 경우 — **거부해도 괜찮지만 순간 버스트는 수용하고 싶은 상황**

## 3. Queue-based Throttling: 줄 세우기

사용자 경험을 헤치지 않으면서 처리량을 조절할 수 있는 가장 직관적인 방법!<br />
거부하는 대신, **줄을 세운다**. 유저를 대기열에 넣고 시스템이 처리할 수 있는 속도로 순서대로 내보낸다.

### 3-1. 왜 "줄"인가

놀이공원의 어트랙션을 생각하면 직관적이다:

- **Rate Limiting** = 정원 초과 시 입장 거부. "내일 다시 오세요."
- **Token Bucket** = 줄 없이 빈자리가 나면 아무나 입장. 운 좋은 사람만 탄다.
- **Queue** = 번호표를 받고 대기. "현재 328번, 당신은 512번, 예상 대기 10분."

유저는 **자신의 위치와 예상 대기 시간을 안다**. 이러한 투명성이 이 방법의 핵심 가치이다. <br />
"언제 될지 모르겠으니 새로고침"이 아니라 "10분이면 되니까 기다리자"로 유저의 행동이 바뀐다.<br />
재시도 폭풍 대신 대기 화면으로 대체된다.

> 이 상황에서 사용자가 할 수 있는 건 단 한가지,<br />
> 줄을 선다. 그리고, 기다린다.

### 3-2. 대기열 아키텍처

<img src="/images/20260402_throttling/queue_architecture.svg" width="640" />

유저의 여정을 따라가면:

1. **대기열 진입** — 유저가 "줄 서기" 버튼을 누르면 Redis Sorted Set에 `{timestamp, userId}`로 저장
2. **순번 조회** — Polling으로 "나는 몇 번째인가?"를 실시간으로 확인
3. **토큰 발급** — 스케줄러가 100ms마다 18명씩 꺼내서 입장 토큰 발급
4. **주문 진입** — 토큰을 들고 주문 API에 접근하면, 관문(Interceptor)이 토큰을 검증하고 통과처리
5. **토큰 소비** — 주문이 완료되면 토큰을 즉시 삭제하여 슬롯을 회수

### 3-3. 줄의 속도: 처리량 산정

대기열의 "줄이 빠지는 속도"는 인프라의 물리적 한계에 의해 결정된다. <br />
감..도 필요하지만 무엇보다 수학으로 명확한 수치를 계산하여 구한다.

```
목표 처리량(TPS) = DB 커넥션 풀 / 평균 처리 시간 × 안전 마진
```

| 파라미터       | 값    | 근거                                                                                      |
| -------------- | ----- | ----------------------------------------------------------------------------------------- |
| DB 커넥션 풀   | 50개  | HikariCP 기준, 중소 서비스 안정 운영 범위. 무작정 늘리면 DB의 Context Switching 부하 증가 |
| 평균 처리 시간 | 200ms | 재고 차감 + 쿠폰 검증 + 주문 저장 + 이벤트 발행. 커머스 P95 Latency 기준                  |
| 안전 마진      | 70%   | GC, 네트워크 지연, 예기치 못한 스파이크 대비. 100% 가동은 Cascading Failure 위험          |

**Target TPS = (50 / 0.2) × 0.7 = 175 TPS**

초당 10,000건이 들어와도 DB에는 175건만 도달한다. 나머지 9,825건은 대기열에서 자기 순서를 기다린다.

### 3-4. Thundering Herd 완화: 마이크로 배칭

175 TPS가 목표라면, 1초에 한 번 175명을 내보내면 될까?

175명이 동시에 `POST /orders`를 호출하면 DB 커넥션 풀 50개를 한순간에 점유한다. 상품 조회, 회원 정보 등 다른 API가 전부 커넥션 대기에 빠진다. 이것이 **Thundering Herd(떼몰이)** 문제다.

대신 1초를 10개 구간으로 나누어 **100ms마다 18명씩** 내보내면 피크가 10배 평탄화된다:

<img src="/images/20260402_throttling/thundering_herd.svg" width="640" />

이전의 한번에 175건을 가져오는 경우보다 DB 커넥션 점유가 균등하게 분산되어 다른 API도 정상적으로 동작할 수 있다.

### 3-5. 왜 Redis Sorted Set인가

대기열 자료구조를 선택할 때, "유저가 실시간으로 자기 순번을 알 수 있어야 한다"는 요구사항이 핵심이었다.

| 자료구조         | 순서 보장        | 순번 조회            | 중복 방지        | 원자적 추출     |
| ---------------- | ---------------- | -------------------- | ---------------- | --------------- |
| Kafka Topic      | O (partition 내) | X (offset != 순번)   | X                | X               |
| Redis List       | O (FIFO)         | **O(N)** — LPOS      | X                | O — RPOP        |
| Redis Sorted Set | O (score)        | **O(log N)** — ZRANK | **O** — Set 특성 | **O** — ZPOPMIN |

`ZRANK`의 **O(log N)** 순번 조회가 결정적이었다. 100만 명이 대기 중이어도 순번 조회는 마이크로초 단위로 응답한다.<br />
Redis List의 `LPOS`는 O(N)이라 대기열이 길어질수록 느려지고, Kafka는 offset 기반이라 "내가 몇 번째"를 알 수 없다.

핵심 Redis 연산 정리:

```
진입:       ZADD NX order:waiting-queue {timestamp} {userId}    → O(log N)
순번 조회:   ZRANK order:waiting-queue {userId}                  → O(log N)
전체 대기:   ZCARD order:waiting-queue                            → O(1)
배치 추출:   ZPOPMIN order:waiting-queue 18                       → O(log N × M)
토큰 발급:   SET order:entry-token:{userId} {UUID} EX 300        → O(1)
토큰 검증:   GET order:entry-token:{userId}                       → O(1)
토큰 소비:   DEL order:entry-token:{userId}                       → O(1)
```

`ZADD NX`의 `NX`는 "이미 존재하면 추가하지 않음"을 의미한다.<br />
유저가 대기열 진입 버튼을 여러 번 눌러도 한 번만 등록되고, 최초 진입 시각의 순번이 보존된다.

### 3-6. 토큰의 생명주기와 보안

대기열에서 나온 유저에게 **입장 토큰**을 발급한다. 이 토큰은 보호 대상 서비스 로직에 접근하기 위한 "출입증"이다.<br />
주문뿐 아니라 수강 신청, 티켓 예매, 선착순 쿠폰 발급 등 — 대기열 뒤에 있는 어떤 서비스든 토큰이 있어야 진입할 수 있다.

<img src="/images/20260402_throttling/token_lifecycle.svg" width="640" />

보안 관점에서 네 가지 정도를 고려할 수 있다:

**1) 토큰 = UUID + userId 바인딩**

토큰 값(UUID)만 맞으면 통과시키면 안 된다. 인증된 `userId`와 Redis에 저장된<br />
`entry-token:{userId}`의 값이 일치하는지 확인해야 한다. 그래야 타인의 토큰을 탈취해도 사용할 수 없다.

**2) TTL 설정**

토큰을 받고 아무 행동도 하지 않는 유저가 Active 슬롯을 영원히 점유하는 것을 방지한다.<br />
TTL이 만료되면 토큰이 자동 삭제되고, 다음 유저에게 기회가 넘어간다.<br />
TTL 길이는 서비스 특성에 따라 결정한다 — 결제 플로우가 긴 커머스라면 5분, 단순 클릭 한 번이면 1분이면 충분할 수 있다.

**3) 서비스 로직 완료 시 즉시 삭제 (동기)**

TTL 만료까지 기다리지 않고 서비스 로직이 성공하면 토큰을 바로 삭제한다. 왜 비동기(이벤트)가 아니라 동기 삭제인가?<br />
— 비동기로 처리하면 삭제까지의 짧은 시간 동안 같은 토큰으로 중복 요청이 가능해진다.<br />
"한 번만 성공해야 하는" 연산(주문, 예매, 발급 등)에서는 동기 삭제가 맞다.

**4) 관문(Interceptor)에서 검증**

서비스 로직(Controller)에 도달하기 전에 Interceptor에서 토큰을 검증한다.<br />
토큰이 없거나 유효하지 않으면 즉시 거부한다. 실제 비즈니스 로직에 도달하기 전에 차단해야 의미 있는 관문이다.

## 4. 세 가지 접근법 비교

|                     | Rate Limiting      | Token Bucket         | Queue                          |
| ------------------- | ------------------ | -------------------- | ------------------------------ |
| **초과 요청 처리**  | 즉시 거부 (429)    | 즉시 거부 (429)      | 대기열에 보관                  |
| **유저 경험**       | "나중에 다시 시도" | "나중에 다시 시도"   | "328번, 예상 대기 2분"         |
| **공정성**          | 없음 (운 기반)     | 없음 (운 기반)       | FIFO 보장                      |
| **버스트 허용**     | 불가               | 가능 (버킷 용량만큼) | 무제한 수용                    |
| **구현 복잡도**     | 낮음               | 중간                 | 높음                           |
| **인프라 비용**     | 카운터 1~2개       | Hash 1개             | Sorted Set + String + 스케줄러 |
| **적합한 시나리오** | API 남용 방지      | 서비스 간 호출 제한  | 한정 자원 선착순               |

### 언제 무엇을 쓰는가

**Rate Limiting**: _"보호"가 목적이고 "대기"가 불필요한 경우_

- 사이트 Bot/크롤러 차단, 공개 API 사용량 제한
- 유저가 429를 "정상"으로 받아들이는 상황

**Token Bucket**: _순간 버스트는 허용하되 평균은 제한해야 할 때_

- API Gateway에서 서비스별 트래픽 쉐이핑
- 마이크로서비스 간 호출 보호

**Queue**: _선착순 **공정성**이 중요한 경우_

- 유저가 **기다릴 의사**가 있는 상황 (쇼핑, 예매, 수강 신청)
- 뒷단 시스템의 처리량이 명확히 제한된 경우 (DB 커넥션 풀, PG 한도)

> 한 가지 전략만 쓰라는 법은 없다. 실무에서는 **조합**하는 경우가 많다. 예를 들어:
>
> - API Gateway에서 **Token Bucket**으로 전체 트래픽을 1차 제한
> - 주문 API 앞에 **Queue**로 한정 자원 보호
> - 외부 PG(결제 대행사) 호출에 **Rate Limiting**으로 계약 한도 준수

## 5. 실전에서 고려할 것들

### 5-1. Graceful Degradation: Redis가 죽으면?

대기열이 Redis에 의존하므로, Redis 장애 시 대응가능한 전략이 필요하다.

| 선택지       | 동작                         | 트레이드오프                     |
| ------------ | ---------------------------- | -------------------------------- |
| A. 전체 차단 | 대기열 진입 불가             | 안전하지만 매출 0                |
| B. 바이패스  | 대기열 없이 직접 주문 허용   | 과부하 위험이 있지만 서비스 유지 |
| C. 로컬 폴백 | 인메모리 Rate Limiter로 전환 | 분산 환경에서 정확도 떨어짐      |

실무에서는 "서비스가 멈추면 안된다" 라는 의견으로 B(바이패스) 를 선택한다고 생각할 수 있다.<br />
실제로 DB connection pool(HikariCP 등)으로 마지막 방어선을 갖출 수 있다고 볼 수 있다.<br />

하지만, 만약 다른 여러 서비스가 연결되어 있다면?<br />
대기열로 인해 모든 서비스에 장애가 발생할 수 있다. 명심해야할 점은 대기열보다 전체 서비스의 유지가 훨씬 중요하다.<br />

실무에서는 대기열이 사망할 경우, A(전체차단) 로 전체 차단 후,<br />
해당 대기 서비스를 다시 살리는 방법이 많이 쓰이는 경우도 있다고 했다.

결국 모든 대응 방식마다 트레이드오프가 발생하지만, 이러한 요소를 미리 고려해두고<br />
서비스 운용 시 미리 방법을 준비할 수 있어야 한다.

### 5-2. 분산 환경에서의 스케줄러

서버 인스턴스가 N대일 때, 각각에서 스케줄러가 돌면 배치 크기가 N배가 된다:

```
서버 A: ZPOPMIN 18   ┐
서버 B: ZPOPMIN 18   ├→ 100ms당 54명 = 540 TPS
서버 C: ZPOPMIN 18   ┘   (설계 기준 175 TPS의 3배!)
```

해결 방법:

- **Redis 분산 락**: `SET scheduler-lock NX EX 1` — 한 인스턴스만 스케줄러 실행
- **리더 선출**: 쿠버네티스 리더 선출 또는 Spring Integration의 `LockRegistry` 활용
- **배치 크기 동적 분배**: `BATCH_SIZE / 인스턴스 수`로 조정

> 보통은 리더 선출 혹은 단일 인스턴스에서 배치를 실행하는게 보편적인 방법이라 볼 수 있다.<br />
> 문제 해결방법은 가장 직관적이고 쉬운 방법이 오히려 빠른 길로 안내한다.

### 5-3. 모니터링: 관측 없는 시스템은 의미가 없다

대기열 시스템을 만들어 놓고 모니터링하지 않으면, 장애를 눈치채지 못한다.<br />
반드시 관측해야 할 지표:

| 지표              | 의미                        | 위험 신호                               |
| ----------------- | --------------------------- | --------------------------------------- |
| Queue Depth       | 대기열에 남은 유저 수       | 계속 증가 → 유입 > 처리 속도            |
| Token Issue Rate  | 초당 토큰 발급 수           | 0으로 떨어짐 → 스케줄러 장애            |
| Token Utilization | 발급 토큰 중 실제 주문 비율 | 급감 → 유저 이탈 또는 주문 실패         |
| Active Slots      | Active Zone의 현재 유저 수  | 한계 초과 → TTL이 너무 길거나 처리 지연 |
| Wait Time p(95)   | 유저 체감 대기 시간         | 급증 → 스케일링 필요                    |

특히 **Token Issue Rate가 0**으로 떨어지면 스케줄러가 죽은 것이다.<br />
대기열에 유저는 계속 쌓이는데 아무도 빠져나가지 못한다. 유저 입장에서는 순번이 영원히 바뀌지 않는 상태.<br />
위 지표의 알람을 걸면 대부분의 치명적 장애를 빠르게 확인하고 대응할 수 있다.

# 결론

서버 개발자의 역할은 무엇이냐

1. 서비스에 필요한 서버 기능 구현
2. 시스템을 안정적이고 효율적인 성능으로 최적화하고 운용하는 역할

이 중 2번에서 우리가 다룰 수 있는 문제와 없는 문제는 명확하게 나눌 수 있다.<br />

평소 서비스 사용량에 따라 필요한 서버 리소스, 예측 불가능한 바이럴된 서비스의 예상 사용자 수..

이를 위한 여러 안전장치 중 서버 스로틀링은 안정성과 경험도 높이는 장치라 볼 수 있다.

서버 스로틀링의 본질은 **"시스템이 처리할 수 있는 만큼만 받겠다"** 이다. 방법의 차이는 **초과분을 어떻게 다루는가**에 있다.

- **Rate Limiting** — 초과 요청을 즉시 거부한다. 시스템은 보호되지만, 유저는 버려진다.
- **Token Bucket** — 순간 버스트를 허용하되 평균 처리량을 제한한다. 유연하지만, 결국 초과분은 거부한다.
- **Queue** — 거부 대신 줄을 세운다. 유저는 순번을 알고, 시스템은 감당할 수 있는 속도로만 처리한다.

어떤 전략이 "정답"인지는 상황에 따라 다르다. <br />
API 남용 방지에는 Rate Limiting이면 충분할 수 있고, 마이크로서비스 보호에는 Token Bucket이 적합하다.<br />
하지만 블랙 프라이데이처럼 **유저가 기다릴 의사가 있고, 공정성이 중요한** 상황에서는.. **줄을 세워야 한다.**

줄을 세운다는 것은 단순한 기술적 선택이 아니다. **"당신의 순서는 보장됩니다"** 라는 서비스와 사용자 간의 신뢰이자 약속이다.<br />
그 약속을 지키기 위해 대기열, 스케줄러, 토큰, 관문 등 안정적인 장치를 준비하고 운용하는 역할이<br />
서버 개발자 라는 직업을 매력적이게 만드는게 아닐까...
