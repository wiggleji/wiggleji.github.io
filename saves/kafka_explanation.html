Chapter 01먼저 비유로 이해하자

복잡한 용어 전에, 우체국과 유튜브의 차이로 핵심을 잡자.
📮
전통 MQ = 우체국 (편지)
보내는 사람(Sender)이 편지를 보내면 → 우체국(Queue)에 쌓이고 → 받는 사람(Receiver)이 가져가면 편지가 사라진다.
한 명이 읽으면 끝. 다른 사람은 못 읽는다.
📺
Kafka = 유튜브 (동영상)
크리에이터(Producer)가 영상을 올리면 → 유튜브(Broker)에 계속 남아있고 → 구독자 A가 봐도, 구독자 B도 볼 수 있다. 각자 "어디까지 봤는지"(offset)를 기억한다.
영상이 삭제될 때까지 누구든, 몇 번이든 볼 수 있다.
이 차이가 모든 것을 결정한다

MQ: 메시지가 소비되면 사라진다 → 1:1 전달이 기본
Kafka: 메시지가 남아있다 → 1:N 전달이 자연스럽다

1:1 vs 1:N — 핵심 차이

MQ — 1:1 기본

메시지 하나 → Consumer 하나가 가져감.
여러 Consumer가 같은 Queue를 보면 → 서로 나눠 가짐 (경쟁)

💡 같은 메시지를 여러 서비스가 받으려면?
→ 메시지를 복사해야 함 (Fanout Exchange 등)

Kafka — 1:N 자연스러움

메시지가 로그에 남아있으니까
Consumer Group A가 읽어도 → 메시지 그대로
Consumer Group B도 읽을 수 있음

💡 새 서비스 추가? → 새 Group ID만 만들면 끝!
복사도, 설정 변경도 불필요

MQ: 소비하면 삭제 (1:1) vs Kafka: 남아서 여러 그룹이 독립 소비 (1:N)

Kafka 전체 구조 — Producer, Broker, Topic, Partition, Offset, Consumer Group

Chapter 03Kafka 용어 사전

Kafka에 나오는 모든 단어를 쉬운 비유와 함께 하나씩 정확히 이해하자.
Producer
메시지를 만들어서 보내는 쪽. Kafka에 데이터를 쓰는 애플리케이션.
MQ에서는 "Sender"라고 부름. Kafka에서는 "Producer"라고 부름.
📺 비유: 유튜브 크리에이터 = 영상을 "올리는" 사람
Consumer
메시지를 읽어가는 쪽. Kafka에서 데이터를 가져오는 애플리케이션.
MQ에서는 "Receiver"라고 부름. Kafka에서는 "Consumer"라고 부름.
📺 비유: 유튜브 시청자 = 영상을 "보는" 사람
Broker
Kafka 서버 한 대. 메시지를 저장하고 전달하는 역할.
보통 3대 이상을 묶어서 클러스터(Cluster)로 운영한다.
📺 비유: 유튜브 서버 = 영상이 실제로 저장되는 곳
Cluster
여러 Broker를 묶은 Kafka 서버 그룹. 데이터를 분산 저장하고, 한 대가 죽어도 서비스를 유지한다.
📺 비유: 유튜브 데이터센터 = 서버 수천 대가 모여있는 곳
Topic
메시지의 카테고리/채널 이름. 예: order-events, payment-results
Producer는 Topic에 쓰고, Consumer는 Topic을 구독해서 읽는다.
📺 비유: 유튜브 채널 = "요리 채널", "게임 채널"
Partition
Topic을 물리적으로 쪼갠 단위. Topic 하나가 여러 Partition으로 나뉜다.
① 순서 보장의 단위 — 같은 Partition 안에서만 순서가 보장된다.
② 병렬 처리의 단위 — Partition이 많을수록 동시에 처리 가능.
📺 비유: 채널의 "재생목록" = 재생목록마다 영상이 순서대로 쌓임
Offset
Partition 안에서 각 메시지에 붙는 순서 번호 (0, 1, 2, 3, …)
Consumer가 "나는 여기까지 읽었다"를 이 번호로 기록한다.
📺 비유: 유튜브 "이어보기" 위치 = "어디까지 봤는지" 기억
Consumer Group
같은 group.id를 공유하는 Consumer들의 팀.
팀 안에서 Partition을 나눠 가짐 (같은 Partition은 팀 내 1명만 읽음).
다른 팀(다른 group.id)은 같은 메시지를 독립적으로 읽을 수 있다.
→ 이것이 1:N 구조의 비밀!
📺 비유: 유튜브 구독 그룹 = 각 팀이 독립적으로 "이어보기" 관리
Segment
Partition 안의 실제 디스크 파일. 일정 크기(1GB 기본) 또는 시간이 지나면 새 파일로 롤링된다.
📺 비유: 영상의 "청크 파일" = 큰 영상을 여러 조각으로 나눠 저장
Leader / Follower
각 Partition에는 Leader(읽기/쓰기 담당) 1개 + Follower(복사본) N개가 있다.
Leader가 죽으면 Follower 중 하나가 새 Leader가 된다.
📺 비유: 원본 서버 + 미러 서버
ISR
In-Sync Replicas — Leader와 데이터가 동기화된 Follower 목록.
Leader가 죽었을 때, ISR 안의 Follower만 새 Leader 후보가 된다.
Controller
클러스터의 관리자 Broker. 파티션의 Leader 선출, 메타데이터 관리를 담당한다.
과거엔 ZooKeeper가 이 역할을 도왔지만, 최신 Kafka는 KRaft 모드로 자체 관리한다.
Retention
메시지를 얼마나 오래 보관할지 설정. 기간(예: 7일) 또는 크기(예: 10GB)로 지정.
보관 기간이 지나면 자동 삭제된다. 이 기간 동안은 Replay(다시 읽기)가 가능.

Chapter 04전통 MQ 구조 이해

RabbitMQ / ActiveMQ의 Sender → Queue → Receiver 구조와 Retry를 먼저 이해하자.

기본 구조: Sender → Queue → Receiver

MQ 기본 흐름: push → Queue(FIFO) → deliver → ACK → 삭제

MQ의 Retry 메커니즘

1
메시지 전달 + 처리

Broker가 Receiver에게 메시지를 push한다. Receiver는 처리를 시도한다.

2
성공 시 → ACK

처리 성공 시 Receiver가 basic_ack()를 보내면 Broker가 메시지를 Queue에서 삭제한다.

3
실패 시 → NACK / Reject

basic_nack(requeue=true)를 보내면 메시지가 Queue의 맨 앞에 다시 들어간다 → 자동 재시도.

4
반복 실패 → DLQ (Dead Letter Queue)

N번 실패하면 DLQ라는 별도 Queue로 이동된다. 나중에 수동으로 확인/재처리.

MQ Retry의 장점

RabbitMQ의 DLX(Dead Letter Exchange), ActiveMQ의 DLQ 등은 브로커 레벨에서 재시도와 실패 처리를 해준다. 별도 코드 없이 설정만으로 retry 횟수, delay, DLQ 라우팅이 가능하다.

MQ에서 1:N을 하려면?

RabbitMQ에서 같은 메시지를 여러 서비스에 보내려면 Fanout Exchange를 사용해야 한다. Exchange가 메시지를 각 서비스의 Queue로 복사해서 넣어준다. 

// RabbitMQ — 3개 서비스에 같은 메시지 전달하려면:
// 1) Fanout Exchange 선언
channel.exchangeDeclare("order-events", "fanout");

// 2) 각 서비스마다 별도 Queue 생성 + 바인딩
channel.queueDeclare("payment-queue", ...);
channel.queueBind("payment-queue", "order-events", "");

channel.queueDeclare("inventory-queue", ...);
channel.queueBind("inventory-queue", "order-events", "");

channel.queueDeclare("notification-queue", ...);
channel.queueBind("notification-queue", "order-events", "");

// → 서비스 추가할 때마다 Queue + Binding 추가 필요

Chapter 05Kafka 전체 그림

Producer → Broker(Cluster) → Consumer. 각 요소가 어떻게 연결되는지 전체 구조를 보자.
Kafka 전체 구조: Producer → Cluster(Brokers + Partitions + Replicas) → Consumer Groups

데이터 흐름 한눈에

1
Producer가 Topic에 메시지를 보낸다

메시지의 Key 값에 따라 어떤 Partition에 들어갈지 결정된다. (Key가 없으면 라운드로빈)

2
Broker의 Leader Partition이 메시지를 저장한다

디스크의 Segment 파일 끝에 append. Follower Partition들이 Leader를 복제(fetch)한다.

3
Consumer가 pull 방식으로 가져간다

MQ는 Broker가 push하지만, Kafka는 Consumer가 자기 속도에 맞게 pull한다. → 소비 속도 차이가 나도 안전.

4
Consumer가 offset을 commit한다

"나는 offset 5까지 읽었다"고 기록. 메시지 자체는 삭제되지 않는다. Retention 기간까지 남아있다.

MQ와의 결정적 차이

MQ (Push): Broker가 "야, 메시지 왔어!" → Receiver에게 밀어넣음 → ACK 받으면 삭제
Kafka (Pull): Consumer가 "나 offset 5부터 줘" → 자기가 가져감 → offset만 갱신, 메시지 그대로

Chapter 06Producer 상세

메시지를 보내는 과정에서 내부적으로 무슨 일이 일어나는지, 단계별로 뜯어보자.

Producer 내부 파이프라인

Producer에서 send()를 호출하면 메시지가 바로 Broker에 가는 게 아니다. 내부에서 여러 단계를 거친다:

1
Serializer (직렬화)

Key와 Value를 byte[]로 변환한다. Kafka는 바이트 배열만 전송하므로, String/JSON/Avro 등을 바이트로 바꿔야 한다.
예: StringSerializer, JsonSerializer, AvroSerializer

2
Partitioner (파티션 결정)

이 메시지가 어떤 Partition에 들어갈지 결정한다.

• Key가 있으면: hash(key) % partition수 → 같은 Key는 항상 같은 Partition
• Key가 없으면: Sticky Partitioner가 배치 단위로 라운드로빈

3
Record Accumulator (버퍼에 모으기)

메시지를 바로 보내지 않고, Partition별로 배치(Batch)에 모은다.
batch.size(기본 16KB)가 차거나, linger.ms(기본 0ms) 시간이 지나면 전송.

💡 linger.ms=5로 설정하면 5ms 동안 모아서 한번에 → 네트워크 효율 UP

4
Sender Thread (네트워크 전송)

별도 스레드가 Batch를 Broker에 전송한다. Broker로부터 응답(ack)을 받으면 완료.

acks — "얼마나 확실히 저장되었나?"

Producer가 Broker에 메시지를 보낸 후, 어느 수준까지 확인하고 "성공"으로 볼 것인지를 결정하는 설정이다.

acks 의미 안전성 속도
0 보내고 확인 안 함. "보냈으니 끝" 유실 가능 가장 빠름
1 Leader에 쓰기 완료만 확인 Leader 장애 시 유실 빠름
all (-1) Leader + 모든 ISR에 복제 완료 확인 가장 안전 느림
실무 기본 세팅

주문/결제처럼 데이터 유실이 안 되는 곳에는:
acks=all + min.insync.replicas=2 + replication.factor=3
→ Broker 1대가 죽어도 데이터 안전!

Idempotent Producer (멱등성 Producer)

네트워크 장애로 같은 메시지가 두 번 전송될 수 있다. Kafka 3.0+에서는 기본으로 enable.idempotence=true가 켜져 있어서, Broker가 중복을 자동 감지하고 한 번만 저장한다.

내부적으로 Producer에게 PID(Producer ID)를 부여하고, 각 메시지에 Sequence Number를 붙여서 중복 여부를 판단한다.

Chapter 07Broker, Cluster, Replication

Kafka 서버(Broker)는 어떻게 묶이고(Cluster), 누가 관리하며(Controller/Coordinator), 데이터를 어떻게 복제(Replication)하는가.

Cluster — Broker가 모이면

Broker는 Kafka 서버 프로세스 1개다. 실무에서는 보통 3대 이상의 Broker를 묶어서 Cluster로 운영한다. 1대가 죽어도 나머지가 서비스를 유지하기 위해서다.

3 Broker Cluster — Partition이 Broker 간 분산, Leader/Follower로 복제

각 구성원의 역할 — Controller vs Coordinator vs Broker

Kafka Cluster 안에는 서로 다른 역할을 하는 3가지 역할이 있다. 이것들이 자주 헷갈린다:

Broker
Kafka 서버 프로세스 1개. 모든 Broker가 하는 기본 일:
• Producer가 보낸 메시지를 디스크에 저장 (자기가 Leader인 Partition에)
• Consumer의 fetch 요청에 메시지를 전달
• Follower인 Partition은 다른 Broker의 Leader를 복제(fetch)
🏢 비유: 회사의 일반 직원 — 각자 맡은 업무(Partition)를 처리
Controller
Broker 중 딱 1대가 맡는 클러스터 관리자 역할. 하는 일:
• Partition의 Leader 선출 — Broker가 죽으면 ISR에서 새 Leader를 뽑음
• 메타데이터 관리 — "어떤 Partition이 어떤 Broker에 있는지" 전파
• Broker 장애 감지 → Failover 처리

Controller Broker가 죽으면? → 남은 Broker 중에서 새 Controller가 선출됨.
🏢 비유: 회사의 팀장 — 직원(Broker)이 퇴사하면 업무 재배분
Group Coordinator
Consumer Group마다 1대의 Broker가 맡는 역할. 하는 일:
• 해당 Consumer Group의 Partition 할당(Rebalancing) 관리
• Consumer의 heartbeat 모니터링 — 죽었는지 살았는지 감시
• offset commit 저장 — __consumer_offsets 내부 Topic에 기록

각 Consumer Group마다 다른 Broker가 Coordinator가 될 수 있다.
🏢 비유: 프로젝트 매니저 — 팀원(Consumer)의 업무(Partition)를 배분하고 상태를 체크
Controller vs Coordinator — 헷갈리지 말자!

Controller = 클러스터 전체를 관리. Partition Leader 선출, Broker 장애 처리. 전체에 1개.
Group Coordinator = Consumer Group을 관리. Rebalancing, heartbeat, offset. Group마다 1개.

Controller는 "Broker 장애 → 파티션 Leader 교체"를 담당하고,
Coordinator는 "Consumer 장애 → 파티션 재할당(Rebalance)"을 담당한다.

KRaft vs ZooKeeper — Controller는 어떻게 선출되나?

ZooKeeper (레거시)

과거에는 ZooKeeper라는 별도 클러스터가 Controller 선출과 메타데이터를 관리했다.

문제: 운영할 시스템이 2개(ZK + Kafka), 장애 포인트 증가, 확장 제한.

KRaft (현재 표준)

Kafka 3.3+부터 KRaft(Kafka Raft) 모드로 Kafka 자체가 Controller를 선출한다.

ZooKeeper 없이 운영 가능. Kafka 4.0에서 ZK 완전 제거. 신규 클러스터는 KRaft 필수.

Replication — 데이터 복제의 전체 흐름

Kafka는 데이터 유실을 방지하기 위해 각 Partition을 여러 Broker에 복제한다.

Leader에서만 읽는다 (쓰기도 읽기도 Leader) Kafka 2.4+: fetch.from.follower 옵션으로 가까운 Follower에서 읽기 가능
Replication: Producer → Leader(쓰기) → Follower가 fetch(복제) → Consumer는 Leader에서 읽기

Leader 장애 시 Failover 과정

1
Controller가 Broker 장애를 감지

Broker의 heartbeat가 끊기면 Controller가 "Broker 1 죽음"을 인지한다.

2
해당 Broker가 Leader였던 모든 Partition에 대해 새 Leader 선출

ISR 목록에서 첫 번째 Follower를 새 Leader로 승격시킨다.

3
메타데이터 업데이트를 모든 Broker에 전파

"Partition 1의 Leader가 Broker 0 → Broker 2로 바뀌었다"는 정보를 전파.

4
Producer/Consumer가 메타데이터를 갱신

새 Leader 정보를 받아서 자동으로 새 Broker에 연결. 애플리케이션 코드 변경 없이 복구.

unclean.leader.election.enable = true 의 위험

ISR이 비어있을 때 (모든 Follower가 느려서 탈락), 동기화 안 된 Follower를 Leader로 선출하면 아직 복제 안 된 데이터가 유실된다.
금융/결제 시스템에서는 반드시 false. 가용성보다 데이터 일관성을 우선.

내구성 보장 공식

// "Broker f대 장애까지 데이터 무손실" 보장 공식:
replication.factor    = f + 1    // 최소 복제본 수
min.insync.replicas   = f + 1    // ISR 최소 유지 수
acks                  = all      // 모든 ISR 확인 후 ACK

// 실무 표준: f=1 (1대 장애까지 허용)
// → replication.factor=3, min.insync.replicas=2, acks=all
//   → Broker 3대 중 1대 죽어도 데이터 무손실!

디스크 저장 구조 (Segment)

각 Partition은 디스크의 디렉토리 하나에 매핑된다. 그 안에 Segment 파일들이 쌓인다:

# Broker 디스크: /kafka-logs/order-events-0/ (Partition 0)
├── 00000000000000000000.log        ← 메시지 데이터 (Segment 0)
├── 00000000000000000000.index      ← offset → 파일 위치 매핑
├── 00000000000000000000.timeindex  ← 시간 → offset 매핑
├── 00000000000000005000.log        ← 다음 Segment (offset 5000~)
├── 00000000000000005000.index
└── leader-epoch-checkpoint

# 파일명 = 해당 Segment의 시작 offset
# .log 파일은 1GB(기본)가 되면 새 Segment로 롤링

왜 디스크인데 빠른가? (4가지 비밀)

Sequential I/O
순차 쓰기만 한다. 파일 끝에 append만 하니까 디스크 헤드가 왔다갔다 안 함.
HDD도 순차 쓰기는 600MB/s+. Random I/O보다 100배 빠르다.
📓 비유: 노트에 순서대로 쓰기 vs 페이지 이리저리 넘기며 쓰기
OS Page Cache
Kafka는 자체 메모리 캐시를 안 만든다. 대신 OS의 Page Cache를 활용.
JVM Heap이 아니라 OS 레벨이라 GC 영향 없음.
📓 비유: 내가 캐시를 관리하는 대신, OS한테 맡기기
Zero-Copy
Consumer에게 데이터 보낼 때 sendfile() 시스템콜 사용.
디스크 → 커널 버퍼 → 네트워크 소켓으로 직접 전송. 애플리케이션 메모리에 복사하지 않음.
📓 비유: 택배 — 창고에서 트럭으로 직접 실음. 사무실에 옮겼다 다시 실는 게 아님.
Batching
메시지를 하나씩 보내지 않고 묶어서(Batch) 보내고, 압축(lz4/snappy/zstd)까지 적용.
네트워크 왕복 횟수와 데이터 크기를 동시에 줄인다.
이 챕터 요약

Broker = Kafka 서버 1대 (저장 + 전달 + 복제)
Cluster = Broker 여러 대의 묶음 (고가용성)
Controller = Cluster 전체 관리자 (Leader 선출, 메타데이터). 전체에 1개.
Group Coordinator = Consumer Group 관리자 (Rebalance, heartbeat, offset). Group마다 1개.
Replication = Leader + Follower 복제. ISR이 장애 복구의 기준.
내구성 공식 = replication.factor=3 + min.insync.replicas=2 + acks=all

Chapter 08Consumer 상세

Consumer Group, offset 관리, rebalancing — Consumer가 가장 복잡하지만 가장 중요하다.

Consumer Group이 1:N을 만드는 원리

Consumer Group 동작: Group 내 분배 (1:1) + Group 간 독립 (1:N)

Consumer는 여러 Topic · 여러 Partition을 동시에 읽을 수 있다

헷갈리기 쉬운 포인트: "1 Partition = 1 Consumer"라는 규칙은 한 방향의 제한일 뿐이다. 반대로 1개의 Consumer가 여러 Partition, 심지어 여러 Topic의 Partition을 동시에 읽을 수 있다.

Consumer 1개가 여러 Topic, 여러 Partition을 동시에 담당할 수 있다

Offset 관리 — "어디까지 읽었나?"

MQ에서는 ACK를 보내면 메시지가 삭제되니 "어디까지 읽었나"를 추적할 필요가 없다. 하지만 Kafka는 메시지가 남아있으니, Consumer가 스스로 "내가 어디까지 읽었는지"를 기록해야 한다. 이것이 offset commit이다.

방식 동작 위험
Auto Commit 5초마다 자동으로 "여기까지 읽었다" 기록 처리 중 장애 → 중복 소비 또는 유실
Manual Commit 내 코드에서 직접 "여기까지 처리 완료" 기록 코드가 복잡해지지만 정확함

Kafka의 Retry — MQ와 뭐가 다른가?

MQ Retry

Broker가 알아서 해줌:
NACK → 자동 재전달
N회 실패 → DLQ로 자동 이동
설정만으로 Retry 가능

Kafka Retry

Consumer가 직접 해야 함:
실패 시 → 코드에서 재시도 로직 작성
or → 별도 Retry Topic으로 보내기
or → Spring Kafka의 @RetryableTopic 사용
코드 수준의 구현 필요 

// Spring Kafka — @RetryableTopic으로 자동 Retry + DLT 구현
@RetryableTopic(
    attempts = 3,
    backoff = @Backoff(delay = 1000, multiplier = 2.0),
    autoCreateTopics = "true"
)
@KafkaListener(topics = "order-events")
fun handleOrder(event: OrderEvent) {
    // 실패하면 자동으로:
    // 1차: order-events-retry-0 (1초 후)
    // 2차: order-events-retry-1 (2초 후)
    // 3차 실패: order-events-dlt (Dead Letter Topic)
    paymentService.process(event)
}

@DltHandler
fun handleDlt(event: OrderEvent) {
    // 3번 다 실패한 메시지 처리
    alertService.notify("주문 처리 실패: \${event.orderId}")
}

Chapter 09Partition Deep Dive

Partition은 Kafka의 모든 것의 단위다 — 순서, 병렬성, 저장, 복제. 제대로 쓰려면 반드시 깊이 이해해야 한다.

Partition이란 정확히 뭔가?

Topic은 논리적 이름이고, 실제 데이터가 저장되는 곳은 Partition이다. 하나의 Topic은 여러 Partition으로 나뉘고, 각 Partition은 독립된 순서 있는 로그 파일이다.

하나의 Topic, 세 개의 독립된 Partition — 각각 자기만의 offset 순서를 가짐

Partition은 어떻게 만들어지는가?

1
Topic 생성 시 Partition 수를 지정

Topic을 만들 때 --partitions 6처럼 개수를 정한다. 이 시점에 Kafka가 Broker들에 Partition을 분배한다.

2
Broker에 분산 배치

예: Broker 3대, Partition 6개 → 각 Broker에 2개씩 Leader Partition이 배치된다. replication.factor=3이면 각 Partition의 복제본도 다른 Broker에 배치.

3
나중에 늘릴 수 있지만, 줄일 수는 없다!

kafka-topics --alter --partitions 12로 늘리기 가능. 하지만 줄이기는 불가능.
이유: 이미 들어있는 데이터를 재분배할 수 없기 때문. → 처음에 잘 정해야 한다.

# Topic 생성 — Partition 6개, Replica 3개
kafka-topics.sh --create \
  --topic order-events \
  --partitions 6 \
  --replication-factor 3 \
  --bootstrap-server localhost:9092

# Partition 수 확인
kafka-topics.sh --describe --topic order-events
# Topic: order-events  PartitionCount: 6  ReplicationFactor: 3
# Partition: 0  Leader: 0  Replicas: 0,1,2  Isr: 0,1,2
# Partition: 1  Leader: 1  Replicas: 1,2,0  Isr: 1,2,0
# ...

# Partition 늘리기 (6 → 12)
kafka-topics.sh --alter \
  --topic order-events \
  --partitions 12 \
  --bootstrap-server localhost:9092
# ⚠ 기존 Key의 Partition 매핑이 깨질 수 있음! (아래 설명)

메시지는 어떤 Partition에 들어가는가?

Producer가 메시지를 보낼 때, Key 값에 따라 Partition이 결정된다. 이것이 순서 보장의 핵심이다.

같은 Key의 메시지는 항상 같은 Partition에 → 해당 유저의 이벤트 순서가 보장됨

순서 보장 — 되는 것과 안 되는 것

Kafka의 순서 보장은 "Partition 내에서만"이다. 이것을 정확히 이해하지 못하면 실무에서 버그가 생긴다.

✓ 순서가 보장되는 경우

같은 Key → 같은 Partition

user-A의 "주문 생성" → "결제 완료" → "배송 시작"
모두 같은 Partition에 들어가니까 이 순서 그대로 Consumer가 읽는다.

→ Key = userId로 설정하면
한 유저의 이벤트들은 반드시 순서대로 처리됨!

✗ 순서가 보장되지 않는 경우

다른 Key → 다른 Partition

user-A의 주문이 P0에, user-B의 주문이 P1에 들어감.
"user-A 주문"과 "user-B 주문" 중 누가 먼저 처리될지 모름.

→ 서로 다른 Key(다른 유저) 간의 글로벌 순서는 보장 안 됨!
→ 보장이 필요하면 Partition 1개만 써야 함 (성능 포기)

위험한 시나리오: Partition 수 변경 시 순서 깨짐

hash("user-A") % 6 = 2 → Partition 2에 있던 user-A의 메시지들
Partition을 12개로 늘리면: hash("user-A") % 12 = 8 → 갑자기 Partition 8로 이동!

기존 Partition 2의 데이터는 그대로인데, 새 메시지는 Partition 8에 쌓임.
→ user-A의 이벤트가 두 Partition에 분산됨 → 순서 보장 깨짐!

해결: Partition 수를 변경하면, 일시적으로 순서가 깨질 수 있다는 것을 인지하고, 가능하면 처음에 충분한 Partition 수를 잡아라.

Partition 수, 어떻게 정하는가?

고려 요소 가이드
원하는 Consumer 병렬도 최대 Consumer 수 이상으로 설정. Partition 수 = 병렬도 상한.
목표 처리량 단일 Partition 처리량 = p msg/s일 때, 필요한 전체 처리량 T라면 → Partition ≥ T/p
향후 확장 나중에 늘릴 수 있지만 Key 매핑이 깨지니, 2~3배 여유를 두고 시작
순서 보장 범위 같은 Key의 순서만 필요하면 Partition 많아도 OK. 글로벌 순서 필요하면 Partition = 1
Broker 수 Partition이 Broker 수보다 많아야 부하가 고르게 분산됨
실무 가이드라인

소규모 시작: Partition 6~12개 (Consumer 2~4개 기준, 3배 여유)
중규모: 30~50개
대규모 (수백만 msg/s): 100개 이상

주의: Partition이 너무 많으면 → Broker 메모리 사용량 증가, Rebalancing 시간 증가, 파일 디스크립터 증가. 필요한 만큼만, 적절한 여유를 두고.

Key 설계 전략

Key를 어떻게 정하느냐가 순서 보장과 부하 분산을 동시에 결정한다.

시나리오 권장 Key 이유
주문/결제 orderId 같은 주문의 생성→결제→배송 이벤트가 순서대로 처리됨
유저 활동 로그 userId 한 유저의 행동이 순서대로 수집됨
IoT 센서 데이터 deviceId 같은 기기의 데이터가 시간순으로 처리됨
로그 수집 (순서 무관) null 라운드로빈으로 골고루 분산 → 최대 처리량
글로벌 순서 필요 고정값 (또는 Partition 1개) 모든 메시지가 한 Partition에 → 순서 보장, 하지만 병렬성 포기
주의: Hot Partition 문제

특정 Key에 메시지가 몰리면 해당 Partition만 과부하가 걸린다 (Hot Partition).
예: key="VIP-user"에 트래픽의 80%가 몰리면 → 하나의 Partition만 과부하.

해결: Key를 좀 더 세분화하거나 (예: userId + eventType), 정말 균등 분산이 필요하면 Custom Partitioner를 구현한다.

한 줄 정리

Partition = 순서의 단위 + 병렬의 단위 + 저장의 단위
같은 Key → 같은 Partition → 순서 보장 ✓  |  다른 Key → 다른 Partition → 순서 보장 ✗

Chapter 10Partition 할당과 Rebalancing

Consumer가 추가/제거될 때 Partition은 어떻게 재분배되는가?

Partition 할당 (Assignment)

Consumer Group에 Consumer가 들어오면, Kafka가 Partition을 배분한다. 이걸 "Assignment"라고 한다.

Consumer 수에 따른 Partition 할당 변화

Rebalancing — Consumer 추가/제거 시

1
트리거 — 새 Consumer 합류, 기존 Consumer 탈퇴, 또는 Partition 수 변경
2
일시 중단 — 모든 Consumer가 소비를 멈추고, Partition 할당을 해제
3
재할당 — 새로운 할당 계획에 따라 Partition을 다시 배분
4
소비 재개 — 각 Consumer가 할당받은 Partition의 마지막 commit된 offset부터 다시 읽기 시작
Rebalance의 문제점

Rebalance 동안 모든 Consumer가 멈춘다 (Stop-The-World).
Consumer가 자주 죽었다 살아나면 → Rebalance가 반복 → "Rebalance Storm"
해결: session.timeout.ms를 넉넉히 (45초), CooperativeStickyAssignor 사용

Cooperative Rebalancing (Kafka 2.4+)

기존: 모든 Partition을 회수 → 재분배 (Stop-The-World)
개선: 변경이 필요한 Partition만 재할당 (Incremental)
→ 나머지 Consumer는 멈추지 않고 계속 소비 가능!

Chapter 11Kafka 생태계

Kafka는 브로커만이 아니다. 주변 도구들까지 알아야 전체 그림이 보인다.
Kafka Connect
외부 시스템(DB, S3, Elasticsearch 등)과 Kafka를 코드 없이 연결해주는 프레임워크.
Source Connector: 외부 → Kafka (예: MySQL 변경사항 → Kafka)
Sink Connector: Kafka → 외부 (예: Kafka → Elasticsearch)
🔌 비유: USB 어댑터 — 서로 다른 기기를 연결해주는 도구
Debezium
DB의 변경사항을 실시간으로 캡처(CDC)하여 Kafka에 전달하는 Source Connector.
MySQL의 binlog, PostgreSQL의 WAL을 읽어서 INSERT/UPDATE/DELETE를 이벤트로 변환.
📹 비유: DB에 달린 CCTV — 모든 변경을 실시간 녹화
Schema Registry
메시지의 스키마(구조)를 중앙 관리하는 서비스. Avro/Protobuf/JSON Schema 지원.
Producer가 보내는 메시지와 Consumer가 기대하는 메시지 형식이 호환되는지 검증한다.
📋 비유: API 문서 — "이 필드는 String이고 필수야"를 강제
Kafka Streams
Kafka에 있는 데이터를 실시간으로 변환/집계하는 라이브러리.
별도 클러스터 없이 애플리케이션에 내장 가능. 윈도우 집계, 조인, 필터 등 지원.
🔧 비유: 컨베이어 벨트 위에서 실시간 가공
ksqlDB
SQL로 Kafka 스트림을 쿼리할 수 있는 데이터베이스.
SELECT * FROM order_events WHERE amount > 10000 같은 실시간 쿼리 가능.
KRaft
Kafka 자체 내장 합의(Consensus) 프로토콜. ZooKeeper를 대체한다.
Kafka 3.3+에서 정식 지원. Kafka 4.0에서 ZooKeeper 완전 제거 예정.

Chapter 12실전 흐름: 주문이 생기면

"주문 생성 → 결제 → 재고 차감 → 알림"이라는 실제 시나리오로 전체 흐름을 따라가보자.

시나리오: 주문 생성 이벤트

1
Order Service (Producer)가 메시지를 보낸다

Key: orderId=ORD-001, Value: {"item":"laptop", "amount":1500000}
Topic: order-events로 전송.
Partitioner가 hash("ORD-001") % 4 = 2 → Partition 2에 들어감.

2
Broker의 Leader Partition이 저장한다

Partition 2의 Leader(예: Broker 1)가 메시지를 Segment 파일에 append.
offset 번호가 붙는다 (예: offset=42).
acks=all이면: Follower들이 복제를 완료한 후에야 Producer에게 "성공" 응답.

3
Consumer Group A: 결제 서비스가 읽는다

group.id="payment-svc"의 Consumer가 Partition 2에서 offset 42를 poll.
결제 처리 완료 → commitSync()로 offset 43을 commit.

4
Consumer Group B: 재고 서비스도 읽는다

group.id="inventory-svc"의 Consumer가 같은 메시지(offset 42)를 독립적으로 poll.
재고 차감 처리. 메시지는 여전히 Partition에 남아있다.

5
Consumer Group C: 알림 서비스도 읽는다

group.id="notification-svc"도 같은 메시지를 읽어서 카카오톡 알림 발송.
→ 1번 produce로 3개 서비스가 각각 독립 소비! (1:N)

같은 걸 MQ로 했다면?

RabbitMQ: Fanout Exchange + 3개 Queue 생성 + 각각 Binding 설정
새 서비스(Group D) 추가 시? → 또 Queue 만들고 Binding 추가

Kafka: 새 서비스가 group.id="new-svc"로 시작하면 끝.
Broker 설정 변경 없이 과거 메시지까지 처음부터 읽을 수 있다 (Replay!)

메시지 실패 시 흐름 비교

MQ 실패 처리

Chapter 14Delivery Semantics

메시지가 몇 번 전달되는가? — At-Most-Once, At-Least-Once, Exactly-Once.
분산 메시징에서 가장 중요한 3가지 보장 수준.

먼저 전체 구조: "잘 받았다"를 어떻게 확인하는가?

Delivery Semantics를 이해하려면, Producer↔Broker와 Consumer↔Broker 사이에서 "메시지를 잘 받았다/처리했다"를 어떤 구조로 확인하는지를 먼저 알아야 한다.

① Producer → Broker: ACK 흐름

Producer가 메시지를 보내면, Broker가 "잘 저장했다"는 응답(ACK)을 돌려준다. 이 ACK의 범위가 acks 설정이다.

Producer의 "잘 받았다" 확인 = acks 설정 (0 / 1 / all)

② Consumer → Broker: Offset Commit 흐름

Consumer가 메시지를 읽고 처리한 후, "여기까지 처리했다"를 Broker에 기록하는 것이 offset commit이다. commit 타이밍에 따라 Delivery Semantics가 결정된다.

Consumer의 "처리했다" 확인 = Offset Commit 타이밍이 Delivery Semantics를 결정
정리: 누가 "잘 받았다"를 확인하는가?

Producer → Broker: acks 설정으로 Broker가 "잘 저장했다" ACK를 보냄. acks 수준이 높을수록 안전하지만 느림.
Consumer → Broker: offset commit으로 Consumer가 "여기까지 처리했다"를 기록. commit 타이밍이 semantics를 결정.

두 구간이 합쳐져서 전체 End-to-End Delivery Semantics가 결정된다:
• At-Most-Once = Producer(acks=0) + Consumer(commit 먼저)
• At-Least-Once = Producer(acks=all) + Consumer(처리 후 commit)
• Exactly-Once = Producer(idempotent + TX) + Consumer(read_committed)

택배 비유로 다시 보기

📦
택배 배송에 비유하면
At-Most-Once (최대 1번) = 택배 기사가 문 앞에 놓고 감. 확인 안 함. 분실되면 끝.
At-Least-Once (최소 1번) = 택배 기사가 수취 확인을 받음. 확인 못 받으면 다시 배송. 가끔 같은 택배가 2번 올 수 있음.
Exactly-Once (정확히 1번) = 택배 기사가 수취 확인 + 중복 방지 시스템까지. 반드시 1번만 도착.

세 가지 보장 수준 상세

❶ At-Most-Once — 최대 1번 (유실 가능, 중복 없음)

의미: 메시지가 0번 또는 1번 전달된다. 유실될 수 있지만, 절대 중복은 아니다.

언제 이렇게 되나?

Producer 측: acks=0으로 보내면 — Broker 응답을 안 기다리니까 네트워크 실패 시 메시지가 사라짐.
Consumer 측: 메시지를 읽자마자 offset commit 먼저 하고, 그 다음 처리하면 — 처리 도중 죽으면 메시지는 "읽었다"고 기록됐으니 다시 안 옴.

유실 OK인 데이터
❷ At-Least-Once — 최소 1번 (유실 없음, 중복 가능)

의미: 메시지가 1번 이상 전달된다. 절대 유실 안 되지만, 같은 메시지가 2번 이상 올 수 있다.

언제 이렇게 되나?

Producer 측: acks=all로 보내는데, Broker가 저장은 했지만 ack 응답이 네트워크에서 유실되면 → Producer가 재전송 → 메시지 2번 저장
Consumer 측: 메시지를 처리한 후 offset commit하는데, commit 전에 죽으면 → 재시작 후 같은 메시지 다시 받음 → 중복 처리

중복 처리 가장 흔한 방식. Consumer에서 멱등성(Idempotency) 처리 필수!
❸ Exactly-Once — 정확히 1번 (유실 없음, 중복 없음)

의미: 메시지가 정확히 1번만 전달되고 처리된다. 이상적이지만 가장 구현하기 어렵고 비용이 크다.

Kafka에서 어떻게 달성하나?

Producer 측: enable.idempotence=true → Broker가 PID + Sequence Number로 중복 감지해서 1번만 저장
전체 흐름: Transactional API → Producer가 "메시지 전송 + offset commit"을 하나의 트랜잭션으로 원자적으로 묶음
Consumer 측: isolation.level=read_committed → commit 완료된 트랜잭션의 메시지만 읽음

세 가지 비교 정리표

항목 At-Most-Once At-Least-Once Exactly-Once
메시지 유실 가능 없음 없음
메시지 중복 없음 가능 없음
Producer 설정 acks=0 acks=all + retries acks=all + idempotence + TX
Consumer 설정 auto commit (처리 전) manual commit (처리 후) read_committed

적합한 상황 로그, 메트릭, 모니터링 주문, 결제, 알림 (대부분) 금융 정산, Kafka Streams

실무에서의 선택

현실적인 조언

대부분의 실무에서는 At-Least-Once + Consumer 멱등성이 표준이다.

왜? Exactly-Once는 Kafka 내부(Topic→Topic)에서만 네이티브로 보장된다. 하지만 실제로 Consumer가 하는 일은 DB에 쓰거나 API를 호출하는 것이다. Kafka TX가 DB TX까지 묶어줄 수는 없다.

따라서 현실적 전략은:
① Producer: acks=all + idempotence=true (중복 전송 방지)
② Consumer: manual commit (처리 후) + 멱등성 처리 (중복 소비 방어)

멱등성(Idempotency) 구현 패턴

At-Least-Once에서 중복 메시지가 와도 결과가 같도록 만드는 3가지 패턴:

// 패턴 1: Processed Events 테이블 (범용적)
@Transactional
fun handleOrder(event: OrderEvent) {
    if (processedEventRepo.existsById(event.eventId)) {
        return  // 이미 처리한 이벤트 → 스킵!
    }
    paymentService.charge(event.orderId, event.amount)
    processedEventRepo.save(ProcessedEvent(event.eventId))
}

// 패턴 2: DB Unique Constraint (가장 간단)
// INSERT 시 중복이면 Duplicate Key Error → catch해서 무시

// UPDATE orders SET status='PAID', version=version+1
// WHERE id=? AND version=?

MQ에서의 Delivery Semantics

RabbitMQ

At-Most-Once: auto-ack=true → 받으면 바로 삭제
At-Least-Once: manual ack + requeue on NACK
Exactly-Once: 네이티브 미지원
→ Consumer 멱등성으로 구현 (Kafka와 동일)

Kafka

At-Most-Once: acks=0 / commit 먼저
At-Least-Once: acks=all / commit 나중
Exactly-Once: Transactional API
→ Kafka 내부 EOS 네이티브 지원

요약 한 줄

At-Most-Once = 빠르지만 유실 감수  |  At-Least-Once = 안전하지만 중복 감수  |  Exactly-Once = 완벽하지만 비용 큼
→ 실무 표준: At-Least-Once + 멱등성

Chapter 15전체 요소 총정리

Kafka의 모든 구성 요소를 역할별로 정리한다.

Producer 측

Serializer
Key/Value를 byte[]로 변환. String, JSON, Avro 등
Partitioner
메시지가 어떤 Partition에 갈지 결정. hash(key) % N
Accumulator
Partition별 배치를 모으는 버퍼. batch.size, linger.ms로 제어
acks
0(확인X), 1(Leader만), all(ISR 전체). 안전성 ↔ 속도 트레이드오프
Idempotence
PID + Sequence Number로 중복 전송 방지. Kafka 3.0+ 기본 on

Broker 측

Topic
메시지 카테고리. 논리적 구분 단위
Partition
Topic의 물리 분할. 순서 보장 + 병렬 처리의 단위
Segment
Partition 내 실제 디스크 파일. .log + .index + .timeindex
Leader / Follower
Partition별 Leader가 R/W, Follower가 복제. ISR이 장애 복구 기준
Controller
Leader 선출, 메타데이터 관리. KRaft(자체) or ZooKeeper(레거시)
Retention
메시지 보관 기간/크기. 이 안에서 Replay 가능

Consumer 측

Consumer Group
같은 group.id의 Consumer 팀. Group 내 분배, Group 간 독립 (1:N의 핵심)
Offset
Partition 내 메시지 순서 번호. Consumer가 "어디까지 읽었나" 기록
Commit
offset을 저장하는 행위. Auto(자동) vs Manual(수동)
Rebalancing
Consumer 증감 시 Partition 재분배. Cooperative 방식으로 중단 최소화
Deserializer
byte[]를 다시 객체로 역직렬화. Producer의 Serializer와 쌍을 이룸

주변 생태계

Kafka Connect
외부 시스템 ↔ Kafka 연결. Source(입력) / Sink(출력) Connector
Debezium
DB CDC → Kafka. binlog/WAL을 실시간 캡처
Schema Registry
메시지 스키마 중앙 관리. Avro/Protobuf/JSON Schema
Kafka Streams
실시간 스트림 처리 라이브러리 (앱 내장)

Delivery Semantics

At-Most-Once
최대 1번. 유실 가능, 중복 없음. acks=0 / commit 먼저. 로그·메트릭용
At-Least-Once
최소 1번. 유실 없음, 중복 가능. acks=all / commit 나중. 실무 표준 (+ 멱등성)
Exactly-Once
정확히 1번. 유실·중복 없음. Transactional API + read_committed. Kafka 내부만 네이티브
Idempotency
멱등성. 같은 메시지를 여러 번 처리해도 결과가 같도록 보장하는 Consumer 측 패턴
최종 기억 공식

Producer → serialize → partition → batch → send → Broker(Leader) → replicate → Follower
Consumer ← poll ← Broker(Leader) ← offset으로 위치 추적 ← commit
Consumer Group: 내부는 1:1 분배, 그룹 간은 1:N 독립 소비
